# Gran Maestro — Best Practices

Gran Maestro를 효과적으로 활용하기 위한 권장 사용 패턴을 정리합니다.

---

## 기본 흐름: 단건 모드

단일 요청을 즉시 시작하고 실시간으로 확인할 때 사용합니다.

```
# 1. 요청을 즉시 시작 (PM 분석 → 스펙 작성 → 자동 실행)
/mst:request 로그인 화면 개선

# 2. 실행 완료 후 결과 수락
/mst:approve
```

**언제 사용하는가:**
- 요청이 1개이고 범위가 명확할 때
- 즉시 결과를 확인하며 진행하고 싶을 때
- 대화형으로 스펙을 조정하며 작업할 때

---

## 베스트 프랙티스: plan 적립 후 일괄 approve

여러 요청을 미리 계획(plan)으로 쌓아두고, 검토 후 한 번에 실행을 시작하는 방식입니다.
자리를 비우기 전에 병렬 작업을 일괄 시작할 때 가장 강력합니다.

```
# 1. 여러 요청을 plan으로 상세화 (스펙 생성)
/mst:plan 로그인 화면 개선
/mst:plan API 엔드포인트 추가
/mst:plan 대시보드 오류 수정

# 2. 생성된 스펙 목록 확인
/mst:list

# 3. 스펙을 검토한 후 일괄 시작 (가상 번호 예시)
/mst:approve PLN-001 PLN-002 PLN-003
```

**언제 사용하는가:**
- 여러 요청을 한 번에 처리하거나 스펙을 미리 준비하고 싶을 때
- 자리를 비우기 전에 여러 작업을 병렬로 시작하고 싶을 때
- 각 스펙을 검토·수정한 후 일괄 승인하고 싶을 때

**장점:**
- 각 스펙을 충분히 검토한 후 실행 시작 가능
- 여러 태스크가 병렬로 동시 실행되어 전체 소요 시간 단축
- 실행 전에 요청 범위를 재조정하거나 취소하기 용이

---

## 단건 vs 일괄 판단 기준

| 상황 | 권장 방식 |
|------|-----------|
| 요청 1개, 즉시 확인 가능 | `/mst:request` 단건 |
| 요청 2개 이상, 미리 준비 | `/mst:plan` + 일괄 approve |
| 요청 범위가 모호함 | `/mst:plan`으로 Q&A 정제 후 시작 |
| 복잡한 작업, 분리 필요 | `/mst:plan`의 분리 실행 기능 활용 |
| 자리를 비울 예정 | `/mst:plan` × N → 일괄 approve → 자리 비우기 |
| 디버그 조사가 선행 필요 | `/mst:debug` → 리포트 확인 → `/mst:plan` |

**핵심 판단 원칙:**
- 요청 범위가 명확하고 즉시 진행한다면 `/mst:request` 단건
- 범위가 불확실하거나 여러 작업을 준비한다면 `/mst:plan`으로 시작

---

## 병렬 실행 최적화

Gran Maestro는 독립적인 태스크를 자동으로 병렬 실행합니다. 설정을 조정하여 최대 처리량을 높일 수 있습니다.

### 병렬 슬롯 수 조정

```
# 기본값(5)보다 높여 더 많은 태스크를 동시 실행
/mst:settings concurrency.max_parallel_tasks 8
```

| 설정 키 | 설명 | 기본값 |
|---------|------|--------|
| `concurrency.max_parallel_tasks` | 동시 실행 가능한 최대 태스크 수 | 5 |

### 병렬 실행 동작 방식

- **독립적인 REQ**: `blockedBy` 의존성이 없으면 자동으로 병렬 실행
- **의존성 있는 REQ**: `blockedBy`에 선행 REQ를 지정하면 선행 완료 후 자동 활성화
- **슬롯 초과 시**: 남은 태스크는 `queued` 상태로 대기하다가 슬롯이 열리면 자동 시작

### 의존성 설정 예시 (plan 작성 시)

```
/mst:plan DB 스키마 변경 → API 엔드포인트 추가 → UI 화면 구현
```

의존성이 있는 연쇄 작업은 `/mst:plan`에서 자동으로 `blockedBy`를 설정합니다.
순서가 중요한 경우 "A 완료 후 B"와 같이 명시적으로 표현하세요.

## 요청 분리 기준

하나의 큰 요청을 여러 REQ로 분리하면 병렬 실행, 독립 리뷰, 부분 롤백이 용이해집니다.

### 분리해야 하는 경우

| 상황 | 이유 | 분리 방법 |
|------|------|-----------|
| 여러 도메인이 혼재 (백엔드 + 프론트엔드) | 각자 독립적으로 테스트/배포 가능 | 도메인별 별도 REQ |
| 순서 의존성이 있는 경우 (DB 변경 → API → UI) | 선행 완료 후 후행 시작 필요 | 분리 + `blockedBy` 설정 |
| 독립적으로 테스트·배포 가능한 경우 | 병렬 실행으로 소요 시간 단축 | 각각 별도 REQ |
| 한 태스크가 지나치게 큰 경우 | 실행 시간 초과 방지, 리뷰 품질 향상 | 기능 단위로 분할 |

### 분리하지 않아도 되는 경우

- 변경 파일 수가 적고 단일 기능에 집중될 때
- 분리 시 오히려 맥락이 흩어져 구현 품질이 떨어질 때
- 원자적(atomic) 변경이 필요한 경우 (동시에 변경하지 않으면 시스템이 깨지는 경우)

### plan의 분리 실행 기능 활용

`/mst:plan` 작성 시 `## 분리 실행` 섹션을 활용하면 PM Conductor가 자동으로 여러 REQ를 생성하고 적절한 `blockedBy`를 설정합니다.

## 자주 쓰는 패턴

### 패턴 1: 오프 전 일괄 실행 (퇴근/자리 비우기)

퇴근 전이나 자리를 비우기 전에 여러 작업을 준비하고 일괄 시작합니다.

```
# 작업 목록을 plan으로 상세화
/mst:plan 로그인 화면 개선
/mst:plan API 엔드포인트 추가
/mst:plan 알림 버그 수정

# 스펙 확인
/mst:list

# 일괄 시작 (가상 번호 예시 — 실제 번호는 /mst:list에서 확인)
/mst:approve PLN-001 PLN-002 PLN-003

# 자리 비우기 — 병렬 실행 중
# 돌아온 후 결과 확인
/mst:list
```

### 패턴 2: 디버그 우선 조사 후 구현

버그를 수정하기 전에 먼저 원인을 파악하고, 분석 결과를 기반으로 구현 계획을 수립합니다.

```
# 1. 버그 원인 조사
/mst:debug 대시보드 새로고침 시 데이터가 초기화되는 문제

# 2. 리포트 확인 (DBG-NNN은 가상 번호)
/mst:inspect DBG-NNN

# 3. 디버그 리포트를 기반으로 구현 계획 수립
/mst:plan --from-debug DBG-NNN

# 4. 스펙 승인 후 실행
/mst:approve
```

**효과:** 원인 분석 없이 구현을 시작하면 엉뚱한 곳을 수정하게 될 위험이 있습니다.
디버그 리포트를 기반으로 스펙을 작성하면 정확한 수정 범위를 보장합니다.

### 패턴 3: 아이디어 검증 후 구현

새로운 기능을 시작하기 전에 여러 AI의 의견을 수집하고 방향을 결정합니다.

```
# 1. 여러 AI에게 아이디어 의견 수집
/mst:ideation 실시간 협업 편집 기능 어떻게 구현할까?

# 2. 종합 의견 검토 후 방향 결정

# 3. 결정된 방향으로 구현 계획 수립
/mst:plan 실시간 협업 편집 기능 구현 (WebSocket 기반)

# 4. 승인 후 실행
/mst:approve
```

**효과:** 방향 결정 없이 바로 구현하면 나중에 대규모 리팩토링이 필요할 수 있습니다.
`/mst:ideation`으로 사전 검증하면 기술 방향을 확정한 후 구현을 시작할 수 있습니다.

### 패턴 4: 복잡한 기능 단계 분리

대형 기능을 여러 단계로 분리하여 점진적으로 구현합니다.

```
# plan 작성 시 분리 실행 섹션 명시
/mst:plan OAuth 소셜 로그인 전체 구현
# → PM Conductor가 자동으로 분리:
#   REQ-NNN-01: DB 스키마 추가 (users.oauth_provider)
#   REQ-NNN-02: 백엔드 OAuth 핸들러 구현 (blockedBy: NNN-01)
#   REQ-NNN-03: 프론트엔드 소셜 로그인 UI (blockedBy: NNN-02)

# 분리된 태스크 확인
/mst:list

# 일괄 승인 (의존성에 따라 순서대로 자동 실행)
/mst:approve REQ-NNN-01 REQ-NNN-02 REQ-NNN-03
```

**효과:** 각 단계를 독립적으로 리뷰·수정할 수 있어 전체 품질이 향상됩니다.
DB 변경이 먼저 완료된 후 API가 시작되므로 불필요한 재작업이 방지됩니다.

### 패턴 5: 세션 중단 후 복구

Claude Code 세션이 종료되었거나 워크플로우가 중단된 경우 복구합니다.

```
# 미완료 요청 목록 확인
/mst:recover

# 특정 요청 복구 (가상 번호 예시)
/mst:recover REQ-042

# 복구 후 현황 확인
/mst:list
```

## 설정 관련 팁

### 주요 설정 키

```
# 병렬 실행 슬롯 조정
/mst:settings concurrency.max_parallel_tasks 8

# 피드백 최대 반복 횟수 조정 (기본 5회)
/mst:settings workflow.max_feedback_rounds 3

# 모든 REQ 완료 시 Maestro 모드 자동 비활성화
# (mode.json의 auto_deactivate 필드로 제어되며, /mst:on 실행 시 기본 true로 설정됨)

# 디버그 로그 활성화
/mst:settings debug.enabled true
/mst:settings debug.log_level debug
```

### 현재 설정 확인

```
/mst:settings
```
